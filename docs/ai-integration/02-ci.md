# 02. CI

CI を「AI が走るゲートの集合」として再設計し、層越境・DIP 違反・ユビキタス言語ドリフトを自動検出する。本章は GitHub Actions 前提で具体例を示す。

---

## 1. CI ゲート全体像

```
PR open
 ├─ (1) install / cache
 ├─ (2) prettier + eslint
 ├─ (3) tsc --noEmit
 ├─ (4) dependency-cruiser  ── 層越境/DIP 違反
 ├─ (5) ubiquitous-language drift check
 ├─ (6) vitest run --coverage
 ├─ (7) bundle size check
 ├─ (8) AI Review (層 / DDD / 命名)
 └─ (9) gate: 上記がすべて green でないとマージ不可
```

AI は **(8) のみ** で走り、(1)〜(7) の決定論的検証の **後段** に立つ。これにより AI のハルシネーションがマージに到達しない。

## 2. 層越境チェック

### 2.1 dependency-cruiser 設定例

```js
// .dependency-cruiser.cjs
module.exports = {
  forbidden: [
    {
      name: "no-domain-to-infra",
      severity: "error",
      from: { path: "^src/domain" },
      to:   { path: "^src/(infrastructure|presentation|application)" },
    },
    {
      name: "no-domain-to-runtime-libs",
      severity: "error",
      from: { path: "^src/domain" },
      to:   { path: "node_modules/(react|axios|node-fetch)" },
    },
    {
      name: "no-presentation-to-infra",
      severity: "error",
      from: { path: "^src/presentation" },
      to:   { path: "^src/infrastructure" },
    },
    {
      name: "no-cycles",
      severity: "error",
      from: {},
      to: { circular: true },
    },
  ],
};
```

### 2.2 GitHub Actions 抜粋

```yaml
- name: Layer boundary check
  run: npx depcruise --config .dependency-cruiser.cjs src
```

### 2.3 違反検出時の AI 補助

depcruise 出力を AI に投入し **修正手順** を生成：

```md
# Task: Fix layer boundary violation

## Input
<depcruise の violations JSON>
<該当ファイル群>

## Output
1. 何が問題か（依存方向の説明）
2. インターフェース抽出案（ファイル新規 / 移動）
3. DI 登録の差分
4. 影響を受けるテストの修正
```

## 3. DIP 違反検知

層越境とは別に、**抽象に依存すべき箇所が具象に依存していないか** を検査する。

### 3.1 ESLint カスタムルール例

```js
// eslint-rules/no-concrete-in-application.js
module.exports = {
  meta: { type: "problem" },
  create(ctx) {
    return {
      ImportDeclaration(node) {
        const file = ctx.getFilename();
        if (!/\/application\//.test(file)) return;
        const src = node.source.value;
        if (/\/infrastructure\//.test(src)) {
          ctx.report({ node, message: "application 層は infrastructure を直接 import 禁止 (DIP 違反)" });
        }
      },
    };
  },
};
```

### 3.2 AI による補完レビュー

ESLint で構造的に検出しきれない（型推論や文字列リテラル経由など）ケースは AI レビューに委ねる：

```md
# Task: DIP audit

## Input
- 変更ファイル一覧と全文

## Check items
- application / domain で具象クラスを new していないか
- domain で I/O API（fetch / axios / fs / window / document）を参照していないか
- DI コンテナ以外で具体クラスを bind していないか

## Output
- 違反箇所と修正案（diff 形式）
- IF 抽出の必要があればその草案
```

## 4. ユビキタス言語ドリフト検知

### 4.1 仕組み

1. `docs/ubiquitous-language.md` を SoT とする
2. CI で `src/domain/**/*.ts` の export 名を抽出
3. 辞典に未登録の名詞を AI が抽出 → エラー
4. 辞典定義と乖離した実装（同名で意味が違う）を AI が指摘

### 4.2 GitHub Actions 抜粋

```yaml
- name: Ubiquitous language drift
  run: |
    node scripts/extract-domain-terms.js > /tmp/terms.json
    npx ts-node scripts/check-ubiquitous-language.ts \
      --terms /tmp/terms.json \
      --dictionary docs/ubiquitous-language.md
```

### 4.3 PR コメント自動投稿

ドリフト検出時、AI が PR コメントに **辞典追加案** を提示：

```md
新規業務用語が検出されました：

| 用語 | 出現箇所 | 推奨定義案 |
| --- | --- | --- |
| `Backorder` | src/domain/ordering/order/backorder.ts | 在庫不足時に確保される将来出荷予定の注文 |

辞典に追加するか、既存用語にリネームしてください。
```

## 5. DDD パターン適合性チェック

### 5.1 ルールセット例

| 指摘項目 | 検出方法 |
| --- | --- |
| Entity に setter が増えた | AST で `set` プロパティ・public setter を検出 → AI に妥当性判定 |
| VO に setter が存在 | AI レビュー（VO は不変） |
| 集約境界外の参照を ID 以外で行っている | AI に Entity の field 型を解析させる |
| Repository が集約以外のクエリを返す | AI レビューで「集約単位での操作」を確認 |
| ドメインイベントが UseCase で発行されている | AI レビューで「Entity が発行・UseCase が publish」を確認 |

### 5.2 AI レビューのプロンプト

```md
# Task: DDD pattern audit

## Inputs
- 変更ファイル全文
- docs/ubiquitous-language.md
- docs/aggregate-map.md

## Audit
1. Entity の不変条件はコンストラクタ/Factory で守られているか
2. VO は不変か（readonly / private set / 再代入なし）
3. 集約は他集約を ID 経由で参照しているか
4. Repository は集約単位か（部分取得・部分保存をしていないか）
5. ドメインイベントの発行点は適切か

## Output
- 指摘（severity: error/warn/info）
- 修正案
```

## 6. コミット & PR 自動化

### 6.1 Conventional Commits 自動生成

```md
# Task: Generate commit message

## Inputs
- git diff --staged
- 直近 10 コミットのメッセージ（スタイル学習用）

## Rules
- type: feat / fix / refactor / test / docs / perf / chore
- scope: 影響レイヤ + コンテキスト (例: domain/ordering)
- subject: 50 字以内、現在形

## Output
<type>(<scope>): <subject>

<body 1-3 行>

BREAKING CHANGE: <あれば>
```

### 6.2 PR 説明文テンプレート

```md
## 概要
<AI が要約>

## 変更レイヤ
- [ ] presentation
- [x] application
- [x] domain
- [ ] infrastructure

## 追加/変更された UseCase
- PlaceOrderUseCase: 在庫チェックを追加

## 破壊的変更
なし

## 手動確認項目
- [ ] /orders/new で在庫切れ商品を選択時、エラー表示
- [ ] 注文確定後の OrderPlaced イベントが publisher に流れる

## 関連
- Issue #123
- ADR-0008
```

### 6.3 リリースノート

ドメインイベント単位でリリースノートを生成すると、業務側にも読みやすい：

```md
# Task: Release notes

## Inputs
- マージ済み PR 一覧
- ドメインイベント一覧（src/domain/**/events.ts）

## Output
- ## 追加された業務シナリオ
- ## 変更された業務シナリオ
- ## 内部改善（非業務影響）
```

## 7. AI レビューの GitHub Actions 例

```yaml
name: AI Review

on:
  pull_request:
    types: [opened, synchronize, reopened]

permissions:
  pull-requests: write
  contents: read

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }

      - name: Get diff
        id: diff
        run: |
          git diff origin/${{ github.base_ref }}...HEAD > /tmp/diff.patch
          echo "size=$(wc -c < /tmp/diff.patch)" >> "$GITHUB_OUTPUT"

      - name: Skip if too large
        if: steps.diff.outputs.size > 200000
        run: echo "Diff too large, skipping AI review."

      - name: AI Review
        if: steps.diff.outputs.size <= 200000
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          node scripts/ai-review.mjs \
            --diff /tmp/diff.patch \
            --rules prompts/system/architecture.md \
            --dictionary docs/ubiquitous-language.md \
            --out /tmp/review.md

      - name: Post review comment
        uses: marocchino/sticky-pull-request-comment@v2
        with:
          path: /tmp/review.md
```

## 8. ゲーティング設計

### 8.1 必須通過ゲート（branch protection）

- `lint`
- `typecheck`
- `dep-boundaries`
- `ubiquitous-language`
- `test`
- `bundle-size`

### 8.2 任意（情報提供）ゲート

- `ai-review`：失敗してもブロックしない（ノイズ防止）
- `mutation-test`：nightly で十分

### 8.3 マージ後

- 自動 ADR ドラフト生成（人間がレビューして merge）
- リリースノート差分蓄積

---

## 9. アンチパターン

| アンチパターン | 対処 |
| --- | --- |
| AI レビューを必須化してコメントが大量＆false positive | severity を分け、error のみ必須化 |
| 全 PR に同一プロンプト | 変更ファイルから関連 ADR・辞典セクションを動的に絞り込んで投入 |
| diff 全体を投入してコスト爆発 | 200KB 超は要約 / chunk 分割。または変更レイヤごとに別ジョブ |
| AI が依存パッケージを勝手に追加 | `package.json` 変更は別ジョブで人間承認必須 |
